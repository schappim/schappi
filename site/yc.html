What is your company going to make? Please describe your product and what it does or will do.

We're building Ninja, a platform that lets developers create, host, and sell tools that AI agents (like ChatGPT, Claude, or custom agents) can use to take real-world actions—e.g. sending emails, querying APIs, or updating CRMs.

These tools run as MCP servers (Model Context Protocol), which expose structured capabilities that agents can safely invoke. Ninja provides a full developer toolchain (CLI, server builder, test harness), a hosting layer (via AWS Lambda), observability (event logs, cost tracing, secrets management), and a marketplace for discoverability.

Agents connect via open standards (MCP), and users can chat with tools, invoke workflows, or build their own. Our goal is to become the Heroku + App Store for AI-native automation.

What tech stack are you using, or planning to use, to build this product? Include AI models and AI coding tools you use.

We're using a Rails monolith for the core web app, with PostgreSQL, Redis, and Sidekiq. Frontend uses Hotwire, Tailwind, and Stimulus.

MCP servers are hosted on AWS Lambda with a nodejs/pythong runtime, supported by a custom CLI toolchain for local dev, deployment, and testing.

For AI, we use:

  • Both OpenAI GPT-4.1 (for remote MCP support) and Anthropic (ported the ClaudeCode SDK to Ruby with the aim of making it so anyone can use Claude Code in the browser to create MCP servers.

Internal tools include a REPL agent that incrementally constructs workflows from natural language, and a Strands-style DAG engine for execution, testing, and observability.

Everything is designed to be agent-native, low-latency, and introspectable — so agents can safely take real-world actions.

Idea

Why did you pick this idea to work on? Do you have domain expertise in this area? How do you know people need what you're making?

I picked this idea out of direct frustration. At Little Bird and Pi Australia, we run a thin-margin electronics business, so I've been automating everything — from shipping labels to warehouse robots. But when I needed to book a plumber, there was no way to get an AI to just do it. It should have been able to check my calendar, negotiate a quote, and send my address — autonomously.

That's when I started experimenting with Model Context Protocol (MCP) tools. I thought it would be easy to layer some structure on top of existing web functionality and generate tools automatically. But the reality was far worse: building MCPs was doable — but hosting, triggering, using, and discovering them sucked.

I have deep domain experience across automation, integration, and developer tooling (15+ years) — and I've now seen dozens of early adopters run into the exact same wall: "I built a tool… now what?" That gap is exactly what Ninja solves.

Who are your competitors? What do you understand about your business that they don't?

Our competitors include:

  • Zapier / Make / n8n – Low-code automation platforms built for humans, not agents. They assume direct user interaction, not autonomous agent execution.
  • LangChain / AutoGen / CrewAI – Great for building agent workflows, but lack production-ready hosting, billing, observability, or marketplace distribution.
  • Vercel / Replit / Modal – Strong infra platforms, but they're app-first, not tool-for-agent-first. None of them natively support agent-triggered execution or the MCP standard.
  • OpenAI GPTs / Anthropic Tools – Closed ecosystems with limited tooling discoverability, no ownership control, and no cross-agent interoperability.

What we understand that they don't:

  1. Agents don't want GUIs—they want affordances.

We're designing tools to be invoked by agents, not through users. That shift in UX assumption changes everything from schema design to invocation flow.

  1. MCPs need a platform, not just a protocol.

Standards like MCP are meaningless without hosting, security, usage tracking, secrets, monetisation, and a discovery layer.

  1. Builders want more than infra—they want distribution.

Developers don't just want to host their tools. They want usage, users, and revenue. That's why we're building the app store and the deployment rails.

  1. THE REAL VALUE IS THE CONTEXT — structured, reusable, and portable.

Ninja is building a context graph that connects users, agents, tools, and data flows. This shared memory layer is what makes agents truly intelligent, auditable, and interoperable across tools. IF YOU ARE BUILDING THE MCP PIPES, YOU GET TO KEEP THE CONTEXT.

In short: we're building the Heroku + App Store for AI agents — not just workflows, not just infra, not just agents. The full stack.

How do or will you make money? How much could you make?

We'll make money in two primary ways:

  1. Revenue share on marketplace usage – When agents invoke tools (MCPs) built by developers, we take a cut of the usage fees (similar to an app store model).

  2. Hosting plans for developers and enterprises – We offer paid tiers for hosting MCPs with added features like secrets management, observability, versioning, and access control.

How much could we make?

There are already zillions of AI agents (GPTs, Claude tools, RAG pipelines, internal copilots) that need structured, secure access to real-world actions. If Ninja powers even a fraction of the agent-to-tool market:

  • With 100,000 active tools doing $50/month in usage, that's $60M/year gross, with Ninja taking a 20–30% cut.
  • Add 20,000 developers on hosting plans at $20–$200/month, and platform revenue could exceed $20–40M/year.

Long-term, if agent usage explodes (as we believe it will), Ninja becomes the default infrastructure layer for tool invocation—like what Stripe did for payments, or Vercel for deployment. The ceiling is extremely high.
